--------------------------------------------------------------------------------
1. SCREEN PURPOSE
--------------------------------------------------------------------------------

Allow user to register a new purchase (received goods from supplier).

Responsibilities:
  • Select provider (supplier) from workspace catalog
  • Select product variant and quantity from provider's catalog
  • Enter pricing (unit price OR line total; compute the other)
  • Optionally override expiry date
  • Validate all required fields
  • On completion:
    - Create Purchase + PurchaseLine records in Dataverse
    - Trigger Power Automate flow: "Purchase Completed → StockBatch creation"
    - Confirm to user; navigate to Inventory view

This screen is the primary entry point for stock inflow into the system.

--------------------------------------------------------------------------------
2. SCREEN ROUTES & FLOWS
================================================================================

Entry Point:
  MainMenu.btnBuy → Navigate(PurchaseEntry)

Exit Routes:
  [Success path]  PurchaseEntry.btnCompletePurchase → Dataverse patch → Flow trigger → Notify → Navigate(StockBatchList)
  [Cancel path]   PurchaseEntry.btnBack → Navigate(MainMenu) [no data saved]
  [Draft path]    (Optional v2) PurchaseEntry.btnSaveDraft → Patch with Status="Draft" → Navigate(BuyList)

Associated Flows:
  • "Purchase Completed → StockBatch Creation" (Power Automate)
    Trigger: PurchaseLine created with parent Purchase.Status = "Completed"
    Actions:
      1. Compute ExpiryDate (manual > lifespan > null)
      2. Create StockBatch record
      3. Return success to canvas app

  • "Normalize Names" (Power Automate)
    Trigger: Provider/Unit/ProductVariant create/update
    Actions: Populate NormalizedName field (lowercase, trim, remove accents)

Screen Display Flow:
  1. User lands on PurchaseEntry with empty form
  2. Selects Provider → provider's catalog loads
  3. Selects ProductVariant → product details populate (unit, last price, etc.)
  4. Enters Qty + UnitPrice → LineTotal auto-computes
  5. Optional: override ExpiryDate
  6. Clicks "Complete Purchase" → validation → patch → flow → success

Error Flow:
  • Missing field → validation toast; no patch sent
  • Dataverse timeout → error toast; button remains active for retry
  • Duplicate purchase (same product+qty) → warning message; allow proceed (no unique constraint)

3. SCREEN STATE & COLLECTIONS
================================================================================

Global Variables (Inherited from gblWorkspaceId):
  gblWorkspaceId: Text (Set by Home screen; REQUIRED)

Screen-Local Variables:
  varSelectedProvider: Record (LookUp result)
    • Current provider selected in dropdown
    • Shape: {ProviderId, Name, NormalizedName}
    • Used to filter ProviderCatalogItem dropdown
    • Blank() if no selection

  varSelectedProduct: Record (LookUp result)
    • Current product/variant selected in dropdown
    • Shape: {ProductVariantId, Name, DisplayName, Unit, LastSellUnitPrice}
    • Used to show product details
    • Blank() if no selection

  varQty: Number
    • Current quantity entered (via cmpQtyStepper)
    • Default: 1
    • Range: >= 0.01 (allow partial units like 0.5 kg)
    • Used to compute total price

  varUnitPrice: Number (Decimal)
    • Price per unit (e.g., 25.50 for $25.50/kg)
    • Optional; if blank, LineTotal must be filled
    • Computed: If varQty and varLineTotal filled, = varLineTotal / varQty
    • On user edit: clears varLineTotal

  varLineTotal: Number (Decimal)
    • Total price for line (e.g., 51.00 for 2 kg × $25.50)
    • Optional; if blank, UnitPrice must be filled
    • Computed: If varQty and varUnitPrice filled, = varQty * varUnitPrice
    • On user edit: clears varUnitPrice

  varExpiryDateManual: DateTime (or blank)
    • Optional override for expiry date
    • If set, overrides ProductFamily.DefaultLifespanDays on flow execution
    • Blank by default

  varBusy: Boolean
    • true while Dataverse patch + flow are executing
    • Disables all buttons; shows spinner
    • Managed by: btnCompletePurchase.OnSelect

  varErrorMessage: Text
    • Displays validation/system errors
    • Cleared on form reset

Collections:
  colProviders: Table
    • Filter(Provider, Workspace = gblWorkspaceId)
    • Items: {ProviderId, Name, NormalizedName, DisplayName}
    • Populated on screen init
    • Sorted: Name ascending

  colProductsInCatalog: Table
    • Filter(ProviderCatalogItem, Provider = varSelectedProvider.ProviderId)
    • Items: {ProductVariantId, ProductVariant.{Name, DisplayName, Unit, LastSellUnitPrice}}
    • Re-populated when varSelectedProvider changes
    • Sorted: ProductVariant.Name ascending

  colUnits: Table
    • Filter(Unit, Workspace = gblWorkspaceId)
    • Items: {UnitId, Name, NormalizedName}
    • Populated on screen init
    • Used for reference/display (not editable in v1)

5. INTERACTION LOGIC (Detailed)
================================================================================

PurchaseEntry.OnVisible:
  // Initialize collections on screen load
  ClearCollect(colProviders,
    Filter(Provider, Workspace = {Value: gblWorkspaceId})
  );
  
  ClearCollect(colUnits,
    Filter(Unit, Workspace = {Value: gblWorkspaceId})
  );
  
  // Reset form fields
  Set(varSelectedProvider, Blank());
  Set(varSelectedProduct, Blank());
  Set(varQty, 1);
  Set(varUnitPrice, Blank());
  Set(varLineTotal, Blank());
  Set(varExpiryDateManual, Blank());
  Set(varBusy, false);
  Set(varErrorMessage, "")

drpProvider.OnChange:
  // User selected a provider
  Set(varSelectedProvider, drpProvider.Selected);
  
  // Clear product selection (not valid for new provider)
  Set(varSelectedProduct, Blank());
  Set(varQty, 1);
  
  // Load provider's catalog
  ClearCollect(colProductsInCatalog,
    Filter(
      ProviderCatalogItem,
      Provider = varSelectedProvider.ProviderId
    )
  );
  
  If(colProductsInCatalog.IsEmpty(),
    Set(varErrorMessage, "No products in this supplier's catalog. Add products first.")
  );
  
  Set(varErrorMessage, "")

drpProduct.OnChange:
  // User selected a product variant
  Set(varSelectedProduct, drpProduct.Selected);
  
  // Suggest last sell price if available
  If(Not IsBlank(drpProduct.Selected.LastSellUnitPrice),
    Set(varUnitPrice, drpProduct.Selected.LastSellUnitPrice);
    Set(varLineTotal, varQty * varUnitPrice)
  );
  
  // If on "Total Price" mode, clear suggested prices (user might prefer total)
  // (Optional: keep last price; let user decide)

cmpQtyStepper.OnChange (from component):
  // Qty changed via stepper
  Set(varQty, event.value);
  
  // Recompute line total if both qty and unit price present
  If(Not IsBlank(varUnitPrice),
    Set(varLineTotal, varQty * varUnitPrice)
  )

cmpMoneyInput.OnChange (UnitPrice input):
  // User entered unit price
  Set(varUnitPrice, event.value);
  
  // Clear line total (user editing unit price)
  Set(varLineTotal, Blank());
  
  // If qty present, compute and show line total
  If(Not IsBlank(varQty) And varQty > 0,
    Set(varLineTotal, varQty * varUnitPrice)
  )

cmpMoneyInput.OnChange (LineTotal input):
  // User entered total price
  Set(varLineTotal, event.value);
  
  // Clear unit price (user editing line total)
  Set(varUnitPrice, Blank());
  
  // If qty present, compute and show unit price
  If(Not IsBlank(varQty) And varQty > 0,
    Set(varUnitPrice, varLineTotal / varQty)
  )

dtpExpiryManual.OnChange:
  // User selected an expiry date
  Set(varExpiryDateManual, dtpExpiryManual.Value)

btnCompletePurchase.OnSelect:
  // Validate all required fields
  Set(varErrorMessage, "");
  
  If(IsBlank(varSelectedProvider),
    Set(varErrorMessage, "Select a supplier.");
    Exit
  );
  
  If(IsBlank(varSelectedProduct),
    Set(varErrorMessage, "Select a product.");
    Exit
  );
  
  If(varQty <= 0,
    Set(varErrorMessage, "Quantity must be greater than 0.");
    Exit
  );
  
  If(IsBlank(varUnitPrice) And IsBlank(varLineTotal),
    Set(varErrorMessage, "Enter unit price or total price.");
    Exit
  );
  
  // Ensure both prices are computed
  If(IsBlank(varUnitPrice),
    Set(varUnitPrice, varLineTotal / varQty)
  );
  
  If(IsBlank(varLineTotal),
    Set(varLineTotal, varQty * varUnitPrice)
  );
  
  // All validation passed; proceed with Dataverse update
  Set(varBusy, true);
  Set(varErrorMessage, "");
  
  // Create Purchase header
  Patch(Purchase,
    Defaults(Purchase),
    {
      Name: "PUR-" & Text(Now(), "yyyymm-dd-hhmmss-" & Rand()),
      Provider: varSelectedProvider,
      Workspace: {Value: gblWorkspaceId},
      PurchaseDate: Now(),
      Status: "Completed"
    }
  ) As newPurchase;
  
  // Create PurchaseLine (this will trigger the flow)
  Patch(PurchaseLine,
    Defaults(PurchaseLine),
    {
      Name: "PLN-" & Text(Now(), "yyyymm-dd-hhmmss-" & Rand()),
      Purchase: newPurchase,
      ProductVariant: varSelectedProduct,
      Qty: varQty,
      UnitPrice: varUnitPrice,
      LineTotal: varLineTotal,
      ExpiryDateManual: varExpiryDateManual,
      Workspace: {Value: gblWorkspaceId}
    }
  );
  
  Set(varBusy, false);
  
  // Success notification
  Notify("Purchase registered. Stock batch will be created in a moment.", NotificationType.Success);
  
  // Clear form for next entry (optional; or navigate away)
  Set(varSelectedProvider, Blank());
  Set(varSelectedProduct, Blank());
  Set(varQty, 1);
  Set(varUnitPrice, Blank());
  Set(varLineTotal, Blank());
  Set(varExpiryDateManual, Blank());
  
  // Navigate to Inventory to see the new batch
  Navigate(StockBatchList)

btnClearForm.OnSelect:
  // Reset all form fields
  Set(varSelectedProvider, Blank());
  Set(varSelectedProduct, Blank());
  Set(varQty, 1);
  Set(varUnitPrice, Blank());
  Set(varLineTotal, Blank());
  Set(varExpiryDateManual, Blank());
  Set(varErrorMessage, "")

Required Tables:

Required Relationships:
  • ProviderCatalogItem.ProductVariant → ProductVariant (1:N)
  • Purchase.Provider → Provider (N:1)
  • PurchaseLine.Purchase → Purchase (1:N)
  • PurchaseLine.ProductVariant → ProductVariant (N:1)

7. ERROR HANDLING & EDGE CASES
================================================================================

Scenario: Provider has no catalog items
  Detected In: drpProvider.OnChange
  Response: Show error message "No products in this supplier's catalog."
  Recovery: User goes to (future) Catalog Management screen to add products to supplier
  Recommendation: In v2, show workflow hint "Add products to this supplier"

Scenario: Qty entered as 0 or negative
  Detected In: cmpQtyStepper or validation on submit
  Response: Show error "Quantity must be > 0" on Submit
  Recovery: User corrects qty in stepper; retry submit
  Note: Stepper should prevent negative values (Min: 0.01)

Scenario: User enters decimal qty (e.g., 0.5 kg)
  Allowed: Yes (AllowDecimals: true on cmpQtyStepper)
  Computation: 0.5 kg × $25/kg = $12.50 (correct)
  Recommendation: Clearly show unit ("0.5 kg") in summary

Scenario: Dataverse patch times out (ISS-021: Error Handling)
  Detected In: Patch() fails; exception thrown
  Response: Show error toast "Failed to save purchase. Retry?"
  Recovery: Button remains active for retry; user clicks again
  Recommendation: Implement exponential backoff (ISS-021 decision pending)

Scenario: User enters Unit Price; then changes Qty
  Expected: Line Total re-computes; Qty × Unit Price = new total
  Actual: cmpQtyStepper.OnChange updates varLineTotal
  Verification: Summary shows updated total

Scenario: User enters Line Total; then changes Qty
  Expected: Unit Price re-computes; Line Total ÷ Qty = new unit price
  Actual: cmpQtyStepper.OnChange updates varUnitPrice
  Verification: Summary shows updated unit price

Scenario: User manually overrides expiry, but ProductFamily has DefaultLifespanDays
  Expected: Flow uses manual override (priority: manual > lifespan > null)
  Actual: PurchaseLine.ExpiryDateManual is passed to flow; flow checks if not null first
  Verification: StockBatch.ExpiryDate = chosen manual date (flow confirms)

Scenario: User selects product; app shows last sell price pre-filled
  Expected: UnitPrice field is pre-populated; LineTotal auto-computes
  Actual: drpProduct.OnChange sets varUnitPrice (if LastSellUnitPrice not null)
  Verification: Qty × LastSellUnitPrice = LineTotal (user can override)

Scenario: Network goes offline mid-form
  Detected In: colProviders or colProductsInCatalog load fails
  Response: Dropdowns show empty or cached data; button submit likely fails with timeout
  Recommendation: Add offline mode or "Retry Load" button (future feature ISS-021)

Scenario: User navigates away (back button) during form entry
  Expected: Form data is lost; no confirmation dialog in v1
  Actual: OnVisible clears all form vars on return
  Recommendation: In v2, implement "Save as Draft" to preserve partial data

8. PERFORMANCE & SCALABILITY
================================================================================

Optimization Notes:
  • colProviders: Filter by Workspace; early in OnVisible (parallel load if possible)
  • colProductsInCatalog: Loaded on drpProvider selection (lazy load; not all at once)
  • No large galleries here; dropdowns are efficient
  • Computations (Qty × Price) are client-side; instant

Load Time Estimate:
  • Screen OnVisible: ~500ms (colProviders load from Dataverse)
  • drpProvider.OnChange: ~300–500ms (colProductsInCatalog load)
  • Patch (Purchase + PurchaseLine): ~1–2s (Dataverse + flow trigger)
  • Flow execution (StockBatch creation): ~5–10s (async; user notified with toast)

Mobile Considerations:
  • Form is vertical (no horizontal scroll)
  • Stepper and money inputs are touch-friendly (min 44px)
  • Dropdown opens full-screen or modal (power apps default)
  • Keyboard appears for numeric inputs; consider numeric keypad for price

Accessibility:
  • All labels associated with inputs
  • Error messages color + text (not just red)
  • All buttons keyboard accessible
  • Stepper buttons have clear +/− icons or text

Scaling:
  • If provider has 1000+ catalog items, dropdown may be slow
    → Recommend: Search filter in dropdown (future v2 feature)
  • If user makes many purchases in one session, no memory leak expected (collections cleared on OnVisible)

11. DESIGN NOTES & OPEN QUESTIONS
================================================================================

Design Approach:
  • Linear form flow (no tabs, step-by-step)
  • Form sections grouped with dividers (visual hierarchy)
  • Two pricing modes (Unit Price or Total) via radio/tab (avoids confusion)
  • Summary box shows data at a glance before submit
  • All buttons at the bottom (standard mobile pattern)

Color & Typography:
  • Labels: Bold, dark gray
  • Info text: Smaller, light gray
  • Error messages: Red background + text
  • Summary: Light gray background (informational)
  • Button colors: Primary (blue for "Complete"); Secondary (gray for "Clear")

Validation UX:
  • Red border on empty required fields (on focus or submit)
  • Tooltip/label shows requirement (e.g., "Qty *")
  • Live validation: Price auto-compute shows user intent understood
  • Error toast on submit failure (not modal; non-blocking)

Open Questions for Next Conversation:
  1. Should pricing mode (Unit Price vs Total Price) be a permanent user preference or per-purchase choice?
  2. Should we show supplier contact info (phone, email) on this screen for reference?
  3. If product is not in provider's catalog, should user be able to add it from here, or must they go to Catalog Management screen first?
